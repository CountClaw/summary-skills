# Repository Guidelines

## 项目结构与内容组织
本仓库是一个总结方法论文档库，核心内容都在 `docs/` 下。`docs/国内/` 与 `docs/国外/` 按来源分类存放方法论文档，文件名使用序号前缀便于排序，例如 `docs/国内/01_总结五步流程模型.md`。`docs/方法总览对比.md` 是总索引，`docs/棱镜总结法PRISM.md` 是综合方法说明。新增或重命名文档时，同步更新总览页中的索引和对比表。

## 构建、校验与本地开发
当前仓库没有构建流程、运行时服务或自动化测试，日常工作以文档校验为主。

```powershell
rg --files docs
rg -n "^#|^##|^###" docs
git diff --check
```

`rg --files docs` 用于确认文档位置；`rg -n "^#|^##|^###" docs` 快速检查标题层级；`git diff --check` 用于发现尾随空格、冲突标记等格式问题。若本地安装了 `markdownlint`，可额外检查 `AGENTS.md` 与 `docs/**/*.md`。

## 编写风格与命名约定
所有内容使用中文撰写，保持说明性、结构化和可复用。沿用现有 Markdown 风格：一级标题说明主题，二级标题组织模块，必要时使用表格、引用和短列表。文件命名遵循 `序号_主题名.md`；综合类文档可使用清晰主题名，如 `棱镜总结法PRISM.md`。避免随意更改既有编号，除非同步调整所有引用。

## 校对与测试要求
本项目没有测试框架，提交前至少完成两项检查：一是人工预览 Markdown 渲染，确认表格、标题和引用块可读；二是检查总览页是否已纳入新增文档。新增方法论时，确认其适用场景、核心特点和一句话描述在总览表中保持一致。

## 提交与 Pull Request 规范
当前仓库尚无可参考的提交历史，建议从现在开始使用简短、明确的提交信息，例如 `docs: 新增 Gibbs 反思循环` 或 `docs: 更新 方法总览对比 表述`。PR 应说明变更目的、影响的文档路径、是否更新了 `docs/方法总览对比.md`；若改动涉及表格排版或大段重构，附一张渲染预览截图更稳妥。
