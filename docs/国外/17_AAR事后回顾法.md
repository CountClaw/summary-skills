# AAR事后回顾法（After-Action Review）

## 概述

AAR（After-Action Review）事后回顾法由**美国陆军**于1970年代开发，最初用于军事行动后的快速复盘，后被广泛应用于企业、项目管理和团队学习中。它用**四个简单的问题**快速完成一次结构化反思。

> 📖 来源：Wharton Executive Education 等多家机构推广

---

## 四个核心问题

| 序号 | 问题 | 目的 |
|------|------|------|
| ① | What was supposed to happen? | 原本计划/期望是什么？ |
| ② | What actually happened? | 实际发生了什么？ |
| ③ | Why was there a difference? | 为什么会有差距？ |
| ④ | What can we learn from this? | 我们能学到什么？ |

---

## 详细说明

### ① 原本计划是什么？（What was supposed to happen?）

**目的：** 建立评估的基准线

**关键问题：**
- 我们当初的目标/计划是什么？
- 期望的结果是什么？
- 关键里程碑和时间节点是什么？

**要点：**
- 回到**当初**的计划，不是事后的"修正版"
- 如果当初没有明确计划，如实记录

**示例：**
> "计划：2周内完成App新版本的UI改版，包含首页、个人中心、设置页三个模块。设计稿需通过产品经理和设计总监双重审核。"

---

### ② 实际发生了什么？（What actually happened?）

**目的：** 客观还原事实

**关键问题：**
- 实际结果是什么？
- 关键事件/转折点是什么？
- 时间线是怎样的？

**要点：**
- **只陈述事实**，不加评判和分析
- 用数据说话
- 不同人可能有不同的视角——都要记录

**示例：**
> "实际情况：用了3周完成。首页和个人中心在第2周完成并通过审核，设置页在第3周才完成。设计稿被设计总监打回2次。产品经理中途变更了个人中心的需求。"

---

### ③ 为什么会有差距？（Why was there a difference?）

**目的：** 深入分析根因

**关键问题：**
- 计划与实际之间的差距在哪？
- 什么导致了这些差距？
- 哪些因素是可控的？哪些是不可控的？

**要点：**
- **对事不对人**
- 区分内部因素（我们可控）和外部因素（我们不可控）
- 不要只看负面差距，也要分析超出预期的正面差距

**分析框架：**
```
内部因素（可控）：
  ✓ 做得好的：提前完成了首页设计
  ✗ 需改进的：没有做好需求变更的缓冲时间

外部因素（不可控但可预测）：
  ✗ 设计总监临时出差，审核延迟3天

外部因素（不可控且不可预测）：
  ✗ 产品经理中途换了人
```

---

### ④ 我们能学到什么？（What can we learn from this?）

**目的：** 提炼可复用的经验

**关键问题：**
- 下次遇到类似情况，我们会有什么不同做法？
- 哪些做法要继续保持？
- 哪些做法需要改进？
- 需要建立什么新规则/流程？

**输出形式：**

| 类型 | 内容 |
|------|------|
| **保持（Keep）** | 首页设计流程有效，继续保持 |
| **改进（Improve）** | 需求变更流程需规范化 |
| **新增（Start）** | 增加"设计审核时间"到项目排期中 |
| **停止（Stop）** | 不允许在开发中途无流程地变更需求 |

## AAR实施指南

### 参与人员

- **所有参与者**都应参加（不只是负责人）
- 需要**安全的环境**——不追责，只学习
- 一名facilitator引导讨论

### 时间建议

| 项目规模 | AAR时长 |
|----------|---------|
| 小任务（1天内） | 15-30分钟 |
| 中等项目（1-4周） | 30-60分钟 |
| 大型项目（1个月+） | 1-2小时 |

### 实施步骤

```
1. 开场（2分钟）
   → 说明目的：学习，不是追责

2. 回答问题①：原本计划（5-10分钟）
   → 对齐所有人的预期

3. 回答问题②：实际发生（10-15分钟）
   → 每个人分享自己的观察
   → 记录事实，不做分析

4. 回答问题③：分析差距（15-30分钟）
   → 集体讨论原因
   → 区分内部/外部因素

5. 回答问题④：总结学习（10-15分钟）
   → 形成Keep/Improve/Start/Stop清单
   → 指定每项的负责人

6. 收尾（2分钟）
   → 感谢参与
   → 确认行动项的跟进时间
```

## AAR的文化要求

AAR的成功关键不在于技巧，而在于**文化**：

| ✅ 需要 | ❌ 避免 |
|---------|---------|
| 安全的环境 | 追责和批评 |
| 全员参与 | 只有领导发言 |
| 诚实面对差距 | 美化结果 |
| 对事不对人 | 找替罪羊 |
| 行动导向 | 只讨论不行动 |
| 及时进行 | 拖到很久之后 |

## 应用场景

| 场景 | 说明 |
|------|------|
| **军事行动** | AAR最初的应用场景 |
| **项目复盘** | 项目结束后的快速复盘 |
| **事件复盘** | 重大事件（成功或失败）后的总结 |
| **季度回顾** | 团队季度绩效回顾 |
| **应急演练** | 演练后的快速评估 |
| **日常事务** | 重要会议/活动后的快速回顾 |

---

## AAR 与其他复盘法的对比

| 方法 | 问题数 | 适用场景 | 特点 |
|------|--------|----------|------|
| **AAR** | 4个 | 快速复盘、团队 | 简洁高效，军事级 |
| **GRAI** | 4步 | 项目复盘、深度 | 更强调规律提炼 |
| **Gibbs** | 6步 | 个人深度反思 | 含情感维度 |
| **KPT** | 3维 | 个人/团队周总结 | 最轻量 |
| **PDCA的C+A** | 嵌入流程 | 持续改进 | 与工作流程合一 |

---

## 一句话总结

> **AAR用四个简单的问题，在15分钟内完成一次高质量的复盘。它的力量不在于复杂，而在于"每次都做、全员参与、对事不对人"。**
